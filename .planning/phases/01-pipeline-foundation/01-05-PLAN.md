**railway.json** — Railway cron + start command (RESEARCH.md Pattern 7):
```json
{
  "$schema": "https://railway.com/railway.schema.json",
  "deploy": {
    "cronSchedule": "0 8 * * *",
    "startCommand": "tsx src/runner.ts"
  }
}
```

Explanation of schedule choice:
- `0 8 * * *` = 08:00 UTC = 09:00 CET (winter) / 10:00 CEST (summer)
- After TED's morning publication batch for German authorities
- Railway cron always interprets schedule as UTC — never use local timezone

**Dockerfile** — pin Node 22 LTS (matches CLAUDE.md requirement):
```dockerfile
FROM node:22-alpine

WORKDIR /app

# Copy dependency manifests first (layer cache: only reinstall when deps change)
COPY package*.json ./

# Install production + dev deps (tsx is a devDep but needed for start command)
RUN npm ci

# Copy source
COPY . .

# /data is NOT created here — it comes from Railway Volume mount (INFRA-02)
# If Volume not attached, runner.ts detects SQLITE_CANTOPEN and exits(2) with message

CMD ["npx", "tsx", "src/runner.ts"]
```

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Railway deployment smoke test</name>
  <what-built>
    All Phase 1 code is complete and committed. Railway deployment configuration is written. The cron job is ready to deploy.
  </what-built>
  <how-to-verify>
    Perform these steps in order:

    **Step 1 — Create a Railway project (if not already done):**
    - Go to https://railway.com → New Project → Deploy from GitHub repo
    - Select the Ausschreibungsskill repo
    - Railway will auto-detect Node.js via package.json

    **Step 2 — Attach a persistent Volume (INFRA-02 — CRITICAL):**
    - In Railway project dashboard → your service → Storage tab → Add Volume
    - Mount path: `/data`
    - This is what persists the SQLite DB across deploys. Without it, scanner.db is wiped every restart.

    **Step 3 — Set environment variables (INFRA-03):**
    - In Railway → service → Variables tab, add:
      ```
      DB_PATH=/data/scanner.db
      ```
    - Phase 2 vars (GMAIL_USER, GMAIL_APP_PASSWORD, ANTHROPIC_API_KEY) can be empty for now

    **Step 4 — Trigger manual deploy:**
    - Railway → Deploy → Deploy Now (or push a commit to trigger auto-deploy)
    - Wait for build to complete (1-2 minutes)

    **Step 5 — Trigger cron manually to test:**
    - In Railway dashboard → your service → there is no "run now" button for cron
    - Instead: temporarily change startCommand to `tsx src/runner.ts` and deploy, or check Railway logs after 08:00 UTC next day
    - Alternative: use Railway CLI: `railway run npx tsx src/runner.ts`

    **Step 6 — Verify in Railway logs:**
    Look for these log lines (all must appear):
    - `[runner] Starting — DB: /data/scanner.db`
    - `[fetcher] Page 1: fetched X, running total X/Y` (X and Y are real numbers > 0)
    - `[fetcher] Total fetched: N / Total available: M` (N should equal or approach M)
    - `[runner] Run complete — available=M fetched=N new=K kept=J dropped=I`
    - `[runner] Exiting with code 0`

    **What to check:**
    - "available" and "fetched" numbers are shown (FETCH-04 — no silent truncation)
    - Process exits cleanly (Railway shows "Completed" not "Failed")
    - Second manual invocation shows `[runner] Another job is already running` OR runs sequentially without overlap
    - Railway Volume shows non-zero size after first run (scanner.db was created)

    **If you see `SQLITE_CANTOPEN` or `ENOENT /data`:**
    - The Volume is not attached. Go to Railway → Storage → attach Volume at /data and redeploy.

    **If you see `[fetcher] Total fetched: 0 / Total available: 0`:**
    - No notices matched the query window. Try adjusting DB_PATH to /tmp/test.db and running with a wider date — or wait for TED to publish the next day's notices.
  </how-to-verify>
